   ```bash
   find "$SCRIPT_DIR/docker-bin" "$SCRIPT_DIR/docker-config" -type f -print | sort | xargs sha256sum
   ```

   ```bash
   find "$SCRIPT_DIR/docker-bin" -type f -print | sort | xargs sha256sum
   ```

3. Remove the CODEX_USER_DIR seeding block (lines 128-131) that copies auth.json -- Codex config is now version-controlled in workspace/.codex/config.toml. However, KEEP the CODEX_USER_DIR variable and its mkdir (line 71) because the compose.yaml volume mount for Codex HOME auth still needs the host directory for credentials (auth.json). Wait -- actually, re-evaluate: the compose.yaml mounts CODEX_USER_DIR to /home/agent/.codex. But now workspace/.codex/config.toml is ALSO mounted at /agent/.codex/config.toml (since workspace mounts at /agent). These are DIFFERENT paths. The codex CLI reads config from ~/.codex/ (which is /home/agent/.codex/) not /agent/.codex/. So:

   Actually, the codex config needs to be at /home/agent/.codex/config.toml for the codex CLI to find it. Currently configure-agent-mcp.sh copies from /opt/docker-config/ to /home/agent/.codex/config.toml. With the new layout, configure-agent-mcp.sh should instead copy from /agent/.codex/config.toml (the workspace version) to /home/agent/.codex/config.toml.

   BUT there is a conflict: compose.yaml mounts CODEX_USER_DIR at /home/agent/.codex, which overwrites whatever is in /home/agent/.codex. The CODEX_USER_DIR mount is for persisting Codex auth (auth.json) across container rebuilds.

   Resolution: Keep the CODEX_USER_DIR volume mount for /home/agent/.codex (it holds auth.json). Change configure-agent-mcp.sh to copy config.toml from /agent/.codex/config.toml (workspace) instead of /opt/docker-config/codex-config.toml. This way auth persists via volume AND config comes from the workspace.

   Keep the codex auth.json seeding lines (129-131) since they handle first-run credential setup. Only remove the codex config template seeding that happened in the Dockerfile.
